## 一、文档目的

## 三、V1 线上观察规则

V1 线上观察期遵循以下规则：

1. 观察对象以真实用户使用反馈为准。
2. 观察范围限定在 V1 已发布能力内。
3. 重点观察 Excel 导入导出、锁定版本、核心测算结果、页面可用性和上线阻断问题。
4. 所有反馈先记录，再判断等级和去向。
5. 不因单个反馈直接进入开发，需先完成分级、影响范围和是否进入 v1.0.1 的判断。
6. V1.0.0 已发布版本不直接替换，不回改已发布 Release。
7. v1.0.1 仅作为 V1 发布后的维护修复版本，不承载 V2 新功能。
8. P4 和明显超出 V1 范围的需求不进入 v1.0.1，统一转入 V2 / V3 需求池。

## 五、问题分级规则

### P0｜线上阻断

符合以下任一情况，判定为 P0：

1. 系统打不开。
2. 登录失败。
3. 项目无法创建。
4. 测算版本无法创建。
5. 保存失败。
6. Excel 导入失败。
7. Excel 导出失败。
8. 锁定版本仍可写入关键数据。

### P1｜核心正确性问题

符合以下任一情况，判定为 P1：

1. 核心测算错误。
2. 收入汇总错误。
3. 成本汇总错误。
4. 税金汇总错误。
5. 利润汇总错误。
6. Excel 导出与页面数据不一致。
7. 锁定版本防护失效。

### P2｜重要体验问题

符合以下任一情况，判定为 P2：

1. 重要体验问题。
2. 导入错误提示不清楚。
3. 表格不好用。
4. 锁定提示不明显。
5. 占位页文案不统一。

### P3｜轻量优化

符合以下情况，判定为 P3：

1. 文案优化。
2. 样式优化。
3. 说明优化。
4. 小交互优化。

### P4｜后续版本需求

以下能力或类似扩展需求判定为 P4：

1. AI。
2. 动态成本。
3. 合约招采。
4. 现金流。
5. 审批流。
6. 多租户。
7. CAD。
8. 移动端。
9. 复杂权限。
10. 指标库大平台。
11. 高级报表。
12. 多项目对比。
13. 多版本差异分析。
14. 其他超出 V1 已发布范围的后续版本需求。

## 六、是否进入 v1.0.1 的判断规则

v1.0.1 作为 V1 发布后的维护修复版本，进入规则如下：

1. P0 必须进入 v1.0.1。
2. P1 必须进入 v1.0.1。
3. P2 可选择进入 v1.0.1，需结合影响范围、修复风险和验证成本判断。
4. P3 可集中进入 v1.0.1，也可延后处理。
5. P4 不进入 v1.0.1，转 V2 / V3 需求池。

## 七、是否转入 V2 需求池的判断规则

符合以下任一情况，建议转入 V2 / V3 需求池：

1. 反馈内容属于 P4。
2. 反馈内容超出 V1 已发布功能范围。
3. 需要新增模块、重构流程、扩展权限体系或引入新的业务对象。
4. 修复会改变 V1 范围冻结结论。
5. 修复会引入明显的数据结构、接口结构或产品范围扩张。
6. 反馈本质是增强能力，而非 V1 既有能力缺陷。

转入 V2 / V3 后，不应以 v1.0.1 修复名义提前开发。

## 八、v1.0.1 发布触发条件

如出现以下情况，可以进入 v1.0.1 修复与发布流程：

1. 出现 P0 线上阻断。
2. 出现 P1 核心测算错误。
3. 真实用户反馈 Excel 导入错误定位困难。
4. 需要将当前维护修复正式交付给用户。
5. 观察期结束后希望沉淀稳定维护版本。

进入 v1.0.1 发布流程后，应另行形成修复清单、验证结论、发布判断，并在确认后再决定是否创建 v1.0.1 tag 和 GitHub Release。
